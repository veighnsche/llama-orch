## Benefits of Contracts

### 1. Clear API Boundaries

Contracts define the interface between components:
- CLI ‚Üî Backend
- UI ‚Üî Backend
- Daemon ‚Üî Daemon

### 2. Versioning

Contracts can be versioned independently:
- `daemon-contract` v1.0.0
- `ssh-contract` v1.0.0
- Breaking changes are explicit

### 3. Documentation

Contracts serve as documentation:
- What types are shared?
- What's the protocol?
- What's stable vs internal?

### 4. Testing

Contracts can be tested independently:
- Serialization/deserialization
- Schema validation
- Protocol compliance

### 5. Code Reuse

Contracts eliminate duplication:
- One definition, many consumers
- DRY principle
- Single source of truth

## Anti-Patterns to Avoid

### ‚ùå Don't Put Everything in Contracts

**Bad:**
```rust
// daemon-contract/src/internal_helper.rs
pub fn internal_helper_function() { ... }  // ‚ùå Internal, not a contract
```

**Good:**
```rust
// daemon-contract/src/handle.rs
pub struct DaemonHandle { ... }  // ‚úÖ Shared type, is a contract
```

### ‚ùå Don't Create Micro-Contracts

**Bad:**
```
97_contracts/
‚îú‚îÄ‚îÄ daemon-handle-contract/  # ‚ùå Too granular
‚îú‚îÄ‚îÄ daemon-status-contract/  # ‚ùå Too granular
‚îî‚îÄ‚îÄ daemon-install-contract/ # ‚ùå Too granular
```

**Good:**
```
97_contracts/
‚îî‚îÄ‚îÄ daemon-contract/  # ‚úÖ Related types together
    ‚îú‚îÄ‚îÄ handle.rs
    ‚îú‚îÄ‚îÄ status.rs
    ‚îî‚îÄ‚îÄ install.rs
```

### ‚ùå Don't Mix Concerns

**Bad:**
```rust
// daemon-contract/src/lib.rs
pub struct DaemonHandle { ... }  // ‚úÖ Contract
pub fn spawn_daemon() { ... }    // ‚ùå Implementation, not contract
```

**Good:**
```rust
// daemon-contract/src/lib.rs
pub struct DaemonHandle { ... }  // ‚úÖ Contract

// daemon-lifecycle/src/lib.rs
pub fn spawn_daemon() { ... }    // ‚úÖ Implementation
```
